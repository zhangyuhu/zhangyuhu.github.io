RTP（Real-time Transport Protocol）

**SSRC（Synchronization Source Identifier）和PT（Payload Type）**是两个核心字段，它们共同确保媒体流的正确传输和解析。

SSRC是一个32位的随机数，用于唯一标识一个RTP流的发送源。

PT是一个8位字段，表示RTP负载的媒体类型（如H.264、Opus、PCMU）。

Payload type: DynamicRTP-Type-96 (96)

Payload type 与 SDP 协商媒体格式不一致

问题是 协商阶段 服务端发送PT=103（H.264），但是在实际传输中使用的确实 PT=96（H.264），导致客户端解码失败无法播放

  • 扩展L: WebRTC 中 RTSP 主要有哪些字段会影响视频播放

    RTSP协议中影响WebRTC视频播放的核心字段包括：

    • 媒体声明（m=video）
    • 编码参数（a=rtpmap、a=fmtp）
    • 控制路径（a=control）
    • SSRC/PT绑定（a=ssrc）
    • 传输扩展（a=rtcp-fb）
    • 安全配置（a=fingerprint）

m=video 554 RTP/AVP 96
m=video：声明媒体类型为视频，WebRTC端需支持对应解码器（如H.264/VP8）: 若声明为m=audio，WebRTC会尝试按音频处理视频流，导致静音且无画面
554：RTSP默认端口，但WebRTC通常通过SETUP请求动态协商传输端口。
RTP/AVP：指定传输协议（如RTP/AVPF需额外协商）

H264/90000：PT=96对应H.264编码，时钟频率90000Hz。
profile-level-id：H.264配置文件（如Baseline/Main/High）。
packetization-mode：分包模式（1=单NAL单元，0=非交错模式）。
若分包模式与接收端预期不符，可能导致RTP包解析错误。

trackID=1：指定RTSP的PLAY请求需携带的URI（如rtsp://example.com/trackID=1）:若控制路径错误，WebRTC端无法启动播放，表现为卡在“连接中”状态

PIPE_DESC_INSTA360_H264_1920_1080_30='v4l2src device=/dev/video0 !  capsfilter name=capsfilter caps= "video/x-h264,width=1920,height=1080,framerate=30/1" ! rtph264pay name=rtph264pay ! webrtcbin name=webrtcbin';

# rtph264pay 指定 pt=103
PIPE_DESC_INSTA360_H264_1920_1080_30='v4l2src device=/dev/video0 !  capsfilter name=capsfilter caps= "video/x-h264,width=1920,height=1080,framerate=30/1" ! rtph264pay name=rtph264pay pt=103 ! webrtcbin name=webrtcbin';

修改后 Payload type 与 SDP 协商值一直，视频可以播放
