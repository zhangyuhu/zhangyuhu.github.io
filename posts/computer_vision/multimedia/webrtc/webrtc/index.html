WebRTC: Web browser Real Time Communication

简介

WebRTC是Google于2011年启动的Web实时通信项目。它是一个开源和免费的项目，为移动应用程序和Web浏览器提供实时通信。它是一种新标准，扩展了Web浏览模型，以点对点的方式处理浏览器。此外，WebRTC标准（如JavaScript API、HTML5、底层通信协议等）由万维网联盟（W3C）和互联网工程任务组（IETF）共同定义，以便在未来的Web浏览器之间建立一个值得信赖的通信通道。本质上，WebRTC允许定义一个API，该API提供对设备上的麦克风和网络摄像头等输入外围设备的安全访问，以点对点的方式与远程设备共享或交换媒体数据、实时数据。

特点：

  • 是基于浏览器的实时音视频（数据）通信技术
  • 免插件
  • 开源
  • 已被W3C纳入HTML5标准
  • 跨平台，跨浏览器，跨移动应用
  • Mac OSX、Windows、iOS、Android、Linux

WebRTC官网https://webrtc.org/，源码的实现网址https://webrtc.googlesource.com/src。

WebRTC 通信过程

基本概念

**STUN (Session Traversal Utilities for NAT) ** 只能UDP，告诉我暴露在广域网的地址IP port ，我通过映射的广域网地址进行P2P数据通信。

**TURN( Traversal Using Relays around for NAT) **UDP或TCP， 打洞失败后，提供服务器中转数据，通话双方数据都通过服务器，占服务器带宽较大 - 为了确保通话在绝大多数环境下可以正常工作。跨网只能用服务器中转（测试发现的） ，使用TURN这种情况在视频通话中占10%

**ICE（Interactive Connectivity Establishment） **即交互式连接建立，是一种用于在不同网络环境下建立对等连接的技术。

  • ICE 的工作原理

    • ICE 的主要目标是让两个端点（通常是浏览器或其他支持 WebRTC 的设备）在各种网络环境下找到彼此并建立直接的连接，以便进行实时数据传输，如音频、视频流或其他应用数据。它通过收集和交换两端的网络候选地址，并进行连接测试，来确定最佳的连接路径。

  • ICE 候选者

    • 类型

      ：ICE 候选者是 ICE 用于建立连接的网络地址和端口信息。主要有以下几种类型。

      • 主机候选者（Host Candidate）：这是本地设备的直接网络地址，通常是设备在局域网内的 IP 地址和端口。例如，在一个家庭网络中，主机候选者可能是设备的私有 IP 地址，如192.168.1.100:5000。
      • 服务器反射候选者（Server - Reflexive Candidate）：当设备位于网络地址转换（NAT）之后时，服务器反射候选者用于获取设备在公网上的地址和端口。ICE 通过向 STUN（Session Traversal Utilities for NAT）服务器发送请求，STUN 服务器会返回设备在 NAT 外部的公网 IP 地址和端口，例如203.0.113.1:6000。
      • 中继候选者（Relay Candidate）：如果设备无法通过主机候选者或服务器反射候选者建立连接，就会使用中继候选者。中继候选者通常是通过 TURN（Traversal Using Relay NAT）服务器进行中转的地址。TURN 服务器会分配一个中继地址，数据会先发送到 TURN 服务器，然后再由 TURN 服务器转发到目标设备。

    • 收集与交换：在 WebRTC 中，RTCPeerConnection对象会自动收集本地的 ICE 候选者。收集完成后，通过信令服务器将本地的 ICE 候选者发送给远端对等体，同时接收远端对等体发送的 ICE 候选者。例如，在之前的代码示例中，通过onicecandidate事件处理函数将本地生成的 ICE 候选者发送给信令服务器，信令服务器再将其转发给远端对等体。

  • NAT 穿透与 ICE

    • NAT 的影响：网络地址转换（NAT）是一种在局域网和公网之间转换 IP 地址的技术，它使得多个局域网内的设备可以共享一个公网 IP 地址。然而，NAT 会导致外部设备难以直接访问局域网内的设备，这给对等连接的建立带来了困难。
    • ICE 如何实现 NAT 穿透：ICE 通过使用 STUN 和 TURN 服务器来实现 NAT 穿透。STUN 服务器帮助设备获取其在 NAT 外部的公网地址，使得对等体之间可以尝试直接连接。如果直接连接失败，TURN 服务器则作为中继，转发数据以绕过 NAT 限制。例如，当两端设备通过 STUN 获取到彼此的公网地址后，会首先尝试直接连接，如果因为 NAT 的限制无法成功，就会借助 TURN 服务器进行数据传输。

  • 连接建立与 ICE

    • 连接测试：在交换 ICE 候选者后，两端设备会对彼此的候选者进行连接测试，以确定最佳的连接路径。测试过程通常是通过发送探测消息来检查候选者之间的可达性。例如，设备 A 会向设备 B 的每个 ICE 候选者发送探测消息，设备 B 收到后会返回响应，通过这种方式来确定哪些候选者可以建立有效的连接。
    • 选择最佳路径：根据连接测试的结果，ICE 会选择一条最佳的连接路径。通常优先选择直接连接（即不通过中继）的路径，如果直接连接不可行，则选择通过 TURN 服务器中继的路径。一旦确定了最佳路径，两端设备就可以通过该路径进行实时数据传输。

ICE 通过收集和交换候选者信息、利用 STUN 和 TURN 服务器实现 NAT 穿透以及进行连接测试和路径选择，使得 WebRTC 能够在复杂的网络环境中建立可靠的对等连接，实现高效的实时通信。

  • WebRTC 终端（两个）：本地和远端，负责音视频采集、编解码、NAT 穿越以及音视频数据传输等；

  • Signal 信令服务器：自行实现的信令服务，负责信令处理，如加入房间、离开房间、媒体协商消息的传递等；

  • STUN/TURN 服务器：负责获取 WebRTC 终端在公网的 IP 地址，以及 NAT 穿越失败后的数据中转服务。

    通信过程如下：

  1. 本地（WebRTC 终端）启动后，检测设备可用性，如果可用后开始进行音视频采集工作；
  2. 本地就绪后，发送“加入房间”信令到 Signal 服务器；
  3. Signal 服务器创建房间，等待加入；
  4. 对端（WebRTC 终端）同样操作，加入房间，并通知另一端；
  5. 双端创建媒体连接对象RTCPeerConnection，进行媒体协商；
  6. 双端进行连通性测试，最终建立连接；
  7. 将采集到的音视频数据通过RTCPeerConnection对象进行编码，最终通过 P2P 传送给对端/本地，再进行解码、展示。

第 6 步在建立连接进行 P2P 穿越时很有可能失败。当 P2P 穿越失败时，为了保障音视频数据仍然可以互通，则需要通过 TURN 服务器进行音视频数据中转。

第二/三/四步：信令交互

什么是信令服务器

信令可以简单理解为消息，在协调通讯的过程中，为了建立一个 webRTC 的通讯过程，在通信双方彼此连接、传输媒体数据之前，它们要通过信令服务器交换一些信息，如加入房间、离开房间及媒体协商等，而这个过程在 webRTC 里面是没有实现的，需要自己搭建信令服务。

媒体协商过程

基本概念

  • SDP（会话描述协议）：是一种用于描述多媒体会话的文本协议，在 WebRTC 中，SDP 用于交换双方的媒体能力、网络信息和编码偏好等内容。

  • Offer/Answer 模型：WebRTC 采用 Offer/Answer 模型进行连接建立和编码协商。一方发送 Offer 消息，其中包含自己支持的编码格式等信息；另一方接收 Offer 后，根据自身能力生成 Answer 消息并返回，以此完成协商。

每端保持两个描述：描述本身的本地描述LocalDescription，描述呼叫的远端的远程描述RemoteDescription。

视频编码协商的具体步骤

步骤一：创建 RTCPeerConnection 对象

在进行视频编码协商之前，需要创建 RTCPeerConnection 对象，它是 WebRTC 连接的核心，负责管理连接的各个方面，包括媒体流的传输和编码协商。

1

const pc = new RTCPeerConnection();

步骤二：发起方生成 Offer

当一方（发起方）想要建立连接时，会调用 createOffer 方法生成一个会话描述（Offer）。这个描述包含了该方支持的媒体类型、编码格式、网络地址等信息。

        return pc.setLocalDescription(offer);

  .catch((error) => {

在 Offer 的 SDP 中，视频编码信息通常以 m=（媒体描述）和 a=rtpmap（RTP 映射）行的形式表示。例如：

m=video 9 UDP/TLS/RTP/SAVPF 96 97
a=rtpmap:96 H264/90000
a=rtpmap:97 VP8/90000

上述 SDP 片段表示发起方支持两种视频编码格式：H.264（负载类型 96）和 VP8（负载类型 97）。

另一方（响应方）接收到发起方的 Offer 后，会调用 setRemoteDescription 方法设置远程描述，然后调用 createAnswer 方法生成自己的会话描述（Answer）。

  .then(() => {

    })
  .then((answer) => {
        return pc.setLocalDescription(answer);
    })
  .then(() => {
        // 将 answer 发送给对方
        sendAnswerToRemote(pc.localDescription);
    })
  .catch((error) => {
        console.error('创建 Answer 时出错:', error);
    });

响应方在生成 Answer 时，会根据自己的支持情况选择合适的编码格式。如果响应方同时支持 H.264 和 VP8，它可以选择其中一种作为协商结果，并在 Answer 中明确表示。例如，如果选择 H.264 编码，Answer 的 SDP 可能如下：

a=rtpmap:96 H264/90000

发起方接收到响应方的 Answer 后，调用 setRemoteDescription 方法设置远程描述。此时，双方的编码协商完成，连接将使用协商好的编码格式进行视频传输。

端与端建立连接

媒体协商结束后，双端统一了传输协议、编解码器等，此时就需要建立连接开始音视频通信了。

但 WebRTC 既要保持音视频通信的质量，又要保证联通性。所有，当同时存在多个有效连接时，它首先选择传输质量最好的线路，如能用内网连通就不用公网，优先 P2P 传输，如果 P2P 不通才会选择中继服务器（relay），因为中继方式会增加双端传输的时长。
